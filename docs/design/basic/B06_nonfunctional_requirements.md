# B06 非機能要件設計書

作成日: 2026-06-07

---

## 1. 匿名性（N01）

**要件**: 個人名と投票内容を紐付けて表示しない。幹事を含め、誰が何に投票したかを特定できない設計とする。

**実現方式**

| 項目 | 方針 |
|------|------|
| 氏名・メールアドレス収集 | 行わない |
| デバイス識別 | localStorageにフラグを保存（`voted_{roomId}: true`）。サーバーには送信しない |
| 投票レコードへのデバイス情報紐付け | 行わない。VoteテーブルにデバイスIDカラムを持たない |
| 投票者の特定可否 | DB上から投票者を特定する手段が存在しない設計とする |

**制約**

- localStorageはブラウザ・デバイス単位のため、同一人物が別デバイスから投票した場合は防止できない。この仕様は許容する（要件定義書F03の「1デバイス1票」に準拠）
- localStorageをクリアすると再投票可能になる。この仕様は許容する

---

## 2. レスポンス性能（N02）

**要件**: 投票後3秒以内に集計結果へ反映する。

**実現方式**

Supabase Realtimeを使用する。投票のINSERTが発生した瞬間にRealtimeイベントが配信され、フロントエンドが受信次第、集計を再取得して画面を更新する。

Supabase無料プランのRealtime上限（同時接続200件）はMVP段階では問題ない。

---

## 3. スマホ対応（N03）

**要件**: iPhoneおよびAndroidのブラウザで利用可能とする。

**実現方式**

| 項目 | 方針 |
|------|------|
| レイアウト | モバイルファースト設計（375px基準） |
| 対応ブラウザ | iOS Safari 16以上 / Android Chrome 110以上 |
| タップターゲット | 最小44×44px（Appleガイドライン準拠） |
| フォントサイズ | 最小16px（iOSの自動ズーム防止） |

---

## 4. セキュリティ（N04）

**要件**: ルームURLはランダムトークンで生成し推測困難とする。個人情報の保存を最小限とする。

**実現方式**

| 項目 | 方針 |
|------|------|
| トークン生成 | UUID v4（Edge Functionでサーバーサイド生成） |
| 参加者トークンと幹事トークン | 別々のUUIDを発行。参加者は幹事URLにアクセスできない |
| RLSポリシー | 参加者トークンで取得できるデータは自ルームのみ（詳細はD02参照） |
| 個人情報 | 氏名・メールアドレス・電話番号は収集しない |
| 通信 | すべてHTTPS（VercelおよびSupabaseが自動対応） |

---

## 5. データ保持（N05）

**要件**: 個人情報の保持期間は適切な期間経過後に自動削除する。

**実現方式**

調査報告書（investigation_report.md）の方針に基づく。

| データ | 削除タイミング |
|--------|--------------|
| ルームデータ | expires_at（開催日時+24時間）から30日後に自動削除 |
| 投票データ | ルームと同時に削除 |

自動削除の実装: Supabase Edge FunctionをCronで定期実行し、削除対象を一括削除する。

---

## 6. SEO・シェア設計（N06）

**要件**: LINEシェア時のOGPを最適化する。飲み会・投票関連キーワードでの検索流入を設計段階から考慮する。

**実現方式**

| 項目 | 方針 |
|------|------|
| OGPタグ | 各画面にog:title / og:description / og:imageを設定 |
| LINEシェア | P02（作成完了画面）にLINEシェアボタンを設置。シェアURLにはOGPが反映される |
| SEOキーワード | 「飲み会 投票」「二次会 アンケート」「帰宅 匿名投票」等を対象に設計（詳細はD07参照） |
