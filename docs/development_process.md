# 開発プロセス定義書
## 飲み会向け匿名投票Webサービス

---

## 1. 開発フロー全体像

```
要件定義（完了）
    ↓
調査課題解決・プライバシー方針確定
    ↓
基本設計（Phase1 MVP）
    ↓
詳細設計（Phase1 MVP）
    ↓
実装（Phase1 MVP）
    ↓
テスト（Phase1 MVP）
    ↓
リリース（Phase1 MVP）
    ↓
基本設計（Phase2）〜 ※以降同様のフェーズを繰り返す
```

各フェーズの成果物は設計書として文書化し、次フェーズの開始前にレビューを完了とする。
Phase2以降は、MVPリリース後に改めて基本設計から同じフェーズを回す。

---

## 2. 基本設計フェーズ

システムの外部仕様を定義する。「何を作るか」を決めるフェーズ。

### 成果物

| # | 設計書名 | 内容 |
|---|---------|------|
| B01 | システム構成設計書 | 技術スタック選定、インフラ構成、全体アーキテクチャ |
| B02 | 画面一覧・画面遷移図 | 全画面の洗い出し、遷移パターンの定義 |
| B03 | ワイヤーフレーム | 各画面のレイアウト・UI要素の配置 |
| B04 | API一覧設計書 | Supabase自動生成エンドポイントの利用方針・RPC一覧・概要定義 |
| B05 | データベース概念設計書 | エンティティ定義、ER図（概念レベル） |
| B06 | 非機能要件設計書 | 匿名性・セキュリティ・パフォーマンスの実現方針 |
| B07 | 広告配置設計書 | 広告を配置するページ・位置・サイズの定義 |

### 留意事項

- B01でフロントエンド・バックエンドの技術スタックおよびブラウザ対応範囲（Safari・Chromeのバージョン等）を確定させてから後続設計に進む
- B06では匿名性の実現方式（Cookieによるデバイス識別など）を明確化する
- B07は設計段階から広告枠を確保し、後付けによるUI崩壊を防ぐ
- プライバシーポリシーの方針（データ保持期間等）は基本設計開始前に確定し、B06に反映する

---

## 3. 詳細設計フェーズ

基本設計を受けて、実装できるレベルまで仕様を落とし込む。「どう作るか」を決めるフェーズ。

### 成果物

| # | 設計書名 | 内容 |
|---|---------|------|
| D01 | 詳細画面設計書 | 各画面の項目定義、バリデーション、エラー表示仕様、ブラウザ対応範囲（Safari・Chrome等バージョン含む） |
| D02 | API詳細設計書 | Supabase自動生成エンドポイントの利用方針・RPC仕様・RLSポリシー定義、リクエスト/レスポンス定義、エラー仕様 |
| D03 | DB物理設計書 | テーブル定義（カラム名・型・制約）、インデックス設計 |
| D04 | シーケンス図 | 主要ユースケースごとのフロントエンド〜バックエンド〜DB間の処理フロー |
| D05 | エラー・例外処理設計書 | エラーパターン一覧、ユーザー向けメッセージ定義 |
| D06 | セキュリティ設計書 | URLトークン生成方式、デバイス識別方式、データ暗号化方針 |
| D07 | OGP・SEO設計書 | メタタグ定義、OGPタグ仕様、ページタイトル・description設計 |

### 主要シーケンス（D04で対象とするフロー）

- ルーム作成〜URL発行
- 参加者のルーム参加〜投票
- 投票結果のリアルタイム集計・反映
- ルームの自動期限処理

---

## 4. 実装フェーズ

詳細設計書に基づき実装する。設計書に記載のない実装は行わない。
設計と実装に乖離が生じた場合は設計書を先に更新する。

### 成果物

| # | 成果物名 | 内容 |
|---|---------|------|
| I01 | コーディング規約 | 命名規則、ファイル構成、コメント規則 |
| I02 | フロントエンド実装 | 各画面の実装 |
| I03 | Supabase設計・実装 | RLSポリシー・マイグレーション・Supabase Functionsの実装 |
| I04 | インフラ構成ファイル | デプロイ設定、環境変数定義 |

---

## 5. テストフェーズ

### 成果物

| # | 設計書名 | 内容 |
|---|---------|------|
| T01 | 単体テスト仕様書 | 関数・コンポーネント単位のテストケース |
| T02 | 結合テスト仕様書 | API〜DB間、フロント〜API間の連携テストケース |
| T03 | システムテスト仕様書 | ユーザーシナリオベースのE2Eテストケース |
| T04 | 負荷・パフォーマンステスト仕様書 | N02（3秒以内の集計反映）の検証シナリオ、同時接続数の上限確認 |
| T05 | テスト結果報告書 | 実施結果、バグ一覧、修正履歴 |

### 優先テストシナリオ

- 匿名性が担保されているか（投票と個人が紐付かないこと）
- 同一デバイスからの重複投票が防止されているか
- リアルタイム集計が3秒以内に反映されるか
- ルームの自動期限が正しく機能するか
- スマートフォン（iOS・Android）での表示・操作

## 6. リリースフェーズ

### 成果物

| # | 成果物名 | 内容 |
|---|---------|------|
| R01 | リリース手順書 | デプロイ手順、ロールバック手順 |
| R02 | 運用・監視設計書 | エラー監視、アクセスログ方針 |
| R03 | プライバシーポリシー | ユーザー向け個人情報取扱方針（方針は基本設計前に確定済み、本文書は公開用文書化） |
| R04 | 利用規約 | サービス利用条件 |

---

## 7. 設計書管理ルール

- 設計書はすべてMarkdownで作成・管理する
- フェーズ完了の条件は「全成果物のレビュー完了」とする
- 設計変更は必ず設計書を更新してから実装に反映する
- 調査課題（要件定義書 第10章）およびプライバシー方針は基本設計フェーズ開始前に解決する

---

## 8. 調査課題の解決タイミング

要件定義書に記載の調査課題は、対応する設計書の作成前に解決する。

| 調査課題 | 解決すべきフェーズ | 対応設計書 |
|---------|----------------|-----------|
| 終電情報API選定 | 基本設計前（Phase1と並行実施） | B01・B04 |
| 個人情報保持期間の法的根拠・プライバシー方針確定 | 基本設計前 | B06・R03 |
| リアルタイム集計の技術方式 | 基本設計 | B01・B04 |
| ルーム有効期限24時間の妥当性 | 基本設計 | B06 |
| ブラウザ対応範囲（Safari・Chrome等バージョン） | 基本設計 | B01・D01 |
